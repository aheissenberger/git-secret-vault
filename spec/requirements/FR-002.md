## Summary

Use a dual-profile vault architecture: authoritative on-disk storage using a content-addressed blob store with a JSONL event log, plus a ZIP interchange format for export/import only.

## Acceptance Criteria

- Authoritative vault storage uses `blobs/<sha256-of-plaintext>.enc`, `index/events.jsonl`, `index/snapshot.json`, and `vault.meta.json`.
- ZIP format (`vault.zip`) is used only for export (`vault zip-export`) and import (`vault zip-import`); it is not the authoritative vault format.
- Core encrypt/decrypt operations do not require external zip/unzip binaries for normal vault operation.
- `vault.meta.json` contains no plaintext filenames or secret paths.

## Verification

- Validate that authoritative vault directory structure matches the expected layout after `init` and `lock`.
- Confirm ZIP export/import roundtrip preserves all entries without data loss.
- Confirm CLI succeeds for normal lock/unlock operations when external zip/unzip tooling is absent.

## Notes

Supersedes the previous ZIP-only vault format description. The ZIP format is retained as an interchange mechanism for interoperability with standard unzip tools via explicit export. See FR-030 through FR-034 for the individual authoritative format components.
