## Summary

Store an encrypted manifest in-vault and enforce deterministic, atomic vault update behavior.

## Acceptance Criteria

- Vault includes encrypted `manifest.json` treated as a normal encrypted entry.
- Manifest stores format version, vault UUID, encryption profile ID, created/updated timestamps, and per-entry metadata (canonical relative path, size, mtime, SHA-256, permissions where applicable, entry type).
- Entry ordering, timestamp normalization policy, and compression settings are deterministic where feasible.
- Determinism does not weaken cryptographic safety.
- Vault writes are atomic using temp-write, best-effort fsync, and rename semantics.

## Verification

- Compare repeated lock operations for deterministic structure behavior.
- Simulate interrupted writes and verify old/new state integrity (no partial vault visibility).
- Validate manifest decrypt/parse contract and required fields.

## Notes

Determinism is constrained by security requirements and platform-specific filesystem behavior.

This is an umbrella requirement. Atomic non-functional decomposition: `NFR-001`, `NFR-002`, `NFR-003`.
