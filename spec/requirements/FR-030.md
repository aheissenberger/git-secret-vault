## Summary

Store each encrypted secret as a content-addressed blob at `blobs/<sha256-of-plaintext>.enc` using XChaCha20-Poly1305 AEAD with a 24-byte random nonce prepended to the ciphertext.

## Acceptance Criteria

- Each vault entry is stored at `blobs/<hex-sha256-of-plaintext-content>.enc`.
- The blob file format is: 24-byte random nonce || XChaCha20-Poly1305 ciphertext || 16-byte Poly1305 tag.
- Blobs are addressed by the SHA-256 hash of the plaintext, enabling content deduplication and integrity verification.
- Writing a blob whose content hash already exists on disk is idempotent (no duplicate write).

## Verification

- Validate that `lock` produces a blob file at the correct content-hash path for each tracked entry.
- Validate blob binary format: bytes 0–23 are the nonce, bytes 24–(N+23) are ciphertext, final 16 bytes are the Poly1305 tag.
- Validate that locking the same plaintext content twice does not create duplicate blob files.

## Notes

Content-addressed storage enables deduplication and supports Git-friendly diffing: blob files change only when secret content changes.
